## 关键陷阱

### 陷阱 1: `question.asked` 是 event 不是 tool call

opencode v1.15.x 早期文档把它描述成 "tool name = question",**但实际它是 Bus 顶层事件**,跟 `message.part.updated` 平级。Nook adapter 在两个地方检测:

1. **PRIMARY 路径**:`handleQuestionAsked` (line ~498),监听顶层 `question.asked` 事件,产 `.waitingForUserInput`。
2. **DEFENSIVE 路径**:`handleToolPart` 里 `if toolName == "question"` 时额外 emit `.waitingForUserInput` (idempotent)。即使 `question.asked` 因为 transport 丢失没到,我们也不会漏判。

两条路径都产 `.waitingForUserInput`,`SessionEvent.determinePhase()` 拿到的 phase 一致。

### 陷阱 2: 早期观察 "v1.17.x 不发 `question.asked`" 是误判

2026-06-17 08:23 测试时观察到 `question.asked` 没到,初步判断 "v1.17.x 不发"。**同一天 08:36 复测时日志清晰显示 `[opencode] event type=question.asked` 正常到达**。根因是 opencode/plugin **socket 启动时序问题**(socket 没绑定),不是事件不存在。

教训:遇到"事件缺失"先看 transport(socket 绑定、plugin 启动顺序),不要急着改 handler 加 fallback。Fallback(陷阱 1 第 2 条)已经写好,作为 defense-in-depth 永远保留,但**不要**因为某个时序问题就把 PRIMARY 路径改成 secondary。

### 陷阱 3: `file.watcher.updated` 高频且无用

v1.17.x 开始高频 emit `file.watcher.updated`(文件系统任何变动)。**Nook plugin 不转发它**,因为它跟 session 状态无关。但用户如果发现 plugin log 里全是这玩意,要知道是 opencode 自己的行为,不是 Nook bug。

## 决策档案

- **`SessionEvent.determinePhase()`** 里 `ToolKind.classify(tool) == .askUserQuestion` → 返回 `.waitingForInput`(不是 `.waitingForApproval`)。`SessionPhase` 加 `isWaitingForInput`(`#58202f5`)。
- **`MCPToolFormatter.toolAliases`** 加 OpenCode 小写映射(`question` → `Question` 等)(`#dadcfa3`)。
- **Adapter 不消费 `file.watcher.updated`** — 留着它只是占带宽。
